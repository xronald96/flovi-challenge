# Shared data model

This is the single source of truth for the `relocation_requests` shape used by both apps. The
database schema (`supabase/schema.sql`) is authoritative; this doc mirrors it in plain language so
each app's own typed model (TS interface in the Vue app, Dart class in the Flutter app) can be
written consistently without sharing runtime code across languages.

## `relocation_requests`

| Field            | Type                            | Notes                                              |
|------------------|----------------------------------|-----------------------------------------------------|
| `id`             | `uuid`                           | primary key, generated by the database              |
| `origin`         | `text`                           | required                                             |
| `destination`    | `text`                           | required                                             |
| `scheduled_date` | `date`                           | required; form validation rejects past dates         |
| `notes`          | `text`                           | optional                                             |
| `status`         | `enum`                           | `pending` \| `booked` \| `completed` \| `cancelled`  |
| `created_by`     | `uuid`                           | references `auth.users.id`; set to the creating user |
| `booked_by`      | `uuid` \| `null`                 | references `auth.users.id`; set when a driver books  |
| `created_at`     | `timestamptz`                    | set by the database                                  |
| `updated_at`     | `timestamptz`                    | auto-updated by a trigger on every row change         |

## Status lifecycle

```
pending --(driver books)--> booked --(dispatcher marks done)--> completed
   \                                                        
    \--(dispatcher cancels)--> cancelled
```

- **Dispatcher app** creates requests as `pending`, and can edit any field including `status`
  (e.g. to `cancelled` or `completed`).
- **Driver app** only ever sees/lists `pending` requests in "Available gigs", and only ever
  transitions a request from `pending` → `booked` (setting `booked_by = auth.uid()`). "My gigs"
  filters to `booked_by = current user`.

## Intentional simplifications

- No separate "dispatcher" vs "driver" role/account type — any authenticated Google account can use
  either app's actions. See `docs/reflection.md` for why, and `supabase/schema.sql` for the RLS
  policies this implies.
- One flat table, no join tables — sufficient for the scope of this challenge.
